# PR 메시지 규칙

## 제목 형식

```
[서비스명] type: 한글 요약 (50자 이내)
```

### 서비스명 (scope)

- 현재 레포의 앱 약어를 사용: `auth`, `base`, `chat`, `llm-gw`, `frontend`, `infra`
- 또는 모듈명으로 구체적으로 지정: `[auth/oauth]`, `[chat/websocket]`
- 여러 서비스가 변경된 경우: `[chat,base]` 형식으로 표기

### type

| type       | 용도                             |
| ---------- | -------------------------------- |
| `feat`     | 새 기능 추가                     |
| `fix`      | 버그 수정                        |
| `refactor` | 리팩토링 (기능 변경 없음)        |
| `perf`     | 성능 개선                        |
| `chore`    | 빌드/설정 변경                   |
| `docs`     | 문서 수정                        |
| `test`     | 테스트 추가/수정                 |
| `style`    | 코드 스타일/포맷팅               |
| `hotfix`   | 긴급 버그 수정 (운영 장애 대응)  |

### 제목 예시

```
[chat] feat: 세무 agent 기능 개선 및 error 처리 추가
[base] fix: 벡터 DB 연결 오류 수정
[auth,chat] refactor: 공통 인증 로직 모듈화
[frontend] feat: 채팅 UI 다크모드 지원
[infra] chore: CI/CD 파이프라인 캐시 설정 추가
```

## 본문 구조

```markdown
## 개요
{이 PR이 무엇을 왜 하는지 2-3문장. 배경 컨텍스트를 포함하여 리뷰어가 빠르게 이해할 수 있도록 작성}

## 주요 변경 내용
- **기능 추가**:
  - {추가된 기능 설명}
- **버그 수정**:
  - {수정된 버그 설명}
- **리팩토링**:
  - {리팩토링 내용 설명}
- **기타**:
  - {그 외 변경 사항}

## 기술 세부사항 (선택)
- {아키텍처 변경, 의존성 추가/변경, 주요 함수/클래스 변경이 있는 경우에만 작성}

## 테스트
- [ ] {테스트 항목 1}
- [ ] {테스트 항목 2}

## 체크리스트
- [ ] 로컬 테스트 완료
- [ ] 문서 업데이트 (필요시)
- [ ] Breaking changes 여부 확인

Closes #{이슈번호}
Related to #{관련 이슈번호}
```

## 각 섹션 작성 가이드

### 개요

- 무엇을 왜 하는지 2-3문장으로 설명
- 배경 컨텍스트를 포함하여 리뷰어가 코드를 보기 전에 동기를 파악할 수 있도록 작성

### 주요 변경 내용

- 커밋 메시지를 단순 나열하지 말고, 카테고리별로 의미 있게 재구성
- 파일 단위가 아닌 기능/논리 단위로 그룹핑
- 해당하는 카테고리만 포함 (변경이 없는 카테고리는 생략)

### 기술 세부사항

- 아키텍처 변경, 의존성 추가/변경, DB 스키마 변경 등이 있을 때만 작성
- 간단한 기능 추가나 버그 수정에서는 생략

### 테스트

- 체크리스트(`- [ ]`) 형식으로 작성
- 수동 테스트와 자동 테스트 항목 모두 포함
- 리뷰어가 직접 검증할 수 있는 구체적인 시나리오 제시

### 이슈 연결

- `Closes #{이슈번호}` 로 이슈 자동 닫기 연결
- 관련은 있지만 이 PR로 완전히 해결되지 않는 이슈: `Related to #{이슈번호}`
- 여러 이슈면 `Closes #12, Closes #34` 형식

### 체크리스트

- 품질 게이트 역할: 머지 전 최소 확인 항목
- 항목은 프로젝트 상황에 맞게 추가/수정 가능

## 작성 프로세스

PR 메시지를 작성할 때 (또는 Claude가 자동 생성할 때) 다음 순서를 따른다:

1. **변경사항 분석**: `git log`, `git diff --stat` 등으로 변경된 파일과 커밋 내용 파악
2. **서비스 식별**: 변경된 파일 경로에서 서비스명 추출 → 제목에 반영
3. **타입 결정**: 커밋 메시지와 변경 내용을 분석하여 가장 대표적인 타입 선택
4. **본문 작성**: 커밋 메시지를 논리적으로 그룹화하고 의미 있는 단위로 재구성

## 출력 형식

PR 제목과 본문은 사용자가 확인/수정할 수 있도록 코드 블록으로 분리하여 제공한다:

**PR 제목:**
```
[chat] feat: 세무 agent 기능 개선 및 error 처리 추가
```

**PR 본문:**
```markdown
## 개요
세무 관련 agent의 기능을 개선하고...
...
```

## 주의사항

- 커밋 메시지를 단순 나열하지 말고, 의미 있게 재구성할 것
- 기술 용어는 한글과 영어를 병행 (예: "벡터 DB (Vector Database)")
- PR 리뷰어가 이해하기 쉽도록 충분한 맥락 제공
- `Closes #` 이슈 연결은 반드시 포함 (이슈 기반 작업인 경우)

## 전체 예시

```
[auth] feat: Google OAuth2 로그인 구현
```

```markdown
## 개요
사용자가 Google 계정으로 로그인할 수 있는 OAuth2 인증 흐름을 구현합니다.
기존 이메일/비밀번호 로그인 외에 소셜 로그인 옵션을 추가하여 가입 전환율을 높이기 위한 작업입니다.

## 주요 변경 내용
- **기능 추가**:
  - Google OAuth2 콜백 엔드포인트 (`/api/v1/auth/google`) 추가
  - GoogleOAuthService 서비스 레이어 구현 (토큰 교환, 사용자 정보 조회)
  - 기존 JWT 발급 로직과 소셜 로그인 연동
- **기타**:
  - OAuth 설정을 위한 환경변수 추가 (`GOOGLE_CLIENT_ID`, `GOOGLE_CLIENT_SECRET`)
  - 관련 단위 테스트 8건 추가

## 테스트
- [ ] Google 로그인 버튼 클릭 → Google 동의 화면 → 콜백 → JWT 발급 확인
- [ ] 이미 가입된 Google 계정으로 재로그인 시 기존 사용자와 매칭 확인
- [ ] 잘못된 OAuth 코드 전달 시 에러 응답 확인

## 체크리스트
- [ ] 로컬 테스트 완료
- [ ] 문서 업데이트 (필요시)
- [ ] Breaking changes 여부 확인

Closes #42
```
